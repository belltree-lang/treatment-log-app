# REBUILD_SCOPE

## 目的

この文書は、`treatment-log-app` の再構築対象を限定するためのスコープ整理です。

前提:

- コード修正はまだ行わない
- 現状機能の優先度と依存境界だけを整理する
- 「どこから触ると壊れやすいか」「どこなら分離しやすいか」を切り分ける

## 今回の結論

再構築対象は、まず以下の 3 領域に限定するのが妥当です。

1. ダッシュボード系
2. 請求系
3. スケジュール候補生成系

施術録本体、勤怠本体、給与本体、Albyte 本体を最初の再構築対象に含めるべきではありません。

## 再構築対象

## A. ダッシュボード系

対象ファイル:

- `src/dashboard/config.gs`
- `src/dashboard/main.gs`
- `src/dashboard/api/*`
- `src/dashboard/auth/*`
- `src/dashboard/data/*`
- `src/dashboard/utils/*`
- `src/dashboard.html`

対象理由:

- ディレクトリとして既に分離されている
- 参照対象シートが比較的限定されている
  - `患者情報`
  - `施術録`
  - `申し送り`
  - `AI報告書`
  - `未回収履歴`
- UI も独立している
- ロール制御、キャッシュ、表示ロジックがまとまっている

再構築時の注意:

- `src/Code.js` 側にもダッシュボード入口が残っている
- `doGet` の責務整理を先に行わないと、移行単位が曖昧になる

## B. 請求系

対象ファイル:

- `src/main.gs`
- `src/get/billingGet.js`
- `src/logic/billingLogic.js`
- `src/logic/billingLogger.js`
- `src/output/billingOutput.js`
- `src/billing.html`
- `src/main.js.html`
- `src/invoice_template.html`

対象理由:

- 既に data / logic / output に分割され始めている
- 対象業務が比較的明確
  - 集計
  - PreparedBilling 保存
  - PDF 生成
  - 請求履歴
  - 入金結果反映
- テストが比較的厚い

再構築時の注意:

- `billingSs()` / `ss()` / `APP.SSID` / Script Properties の依存解決がまだ混在
- UI と保存仕様が強く結合している
- Drive / PDF / 請求履歴への副作用が多い

## C. スケジュール候補生成系

対象ファイル:

- `src/schedule.js`
- `src/intake.html`
- `src/intake_list.html`
- 関連する intake 系関数がある箇所

対象理由:

- 別スプレッドシート依存で、境界が比較的明確
- 候補生成ロジックが独立している
- 住所ジオコーディングや Waitlist/SlotMaster/Assignment など、閉じた業務フローになっている

再構築時の注意:

- Intake 全体が完全分離ではない
- `src/Code.js` 側の受付ロジックと接続している部分は要調査
- `validateSlotMaster()` のような重複定義があるため、先に責務整理が必要

## 再構築対象外

## 1. 施術録本体

対象外ファイル:

- `src/Code.js` の施術録関連
- `src/app.html`
- News / 申し送り / AI報告書の密結合部分

対象外理由:

- 患者ヘッダ取得、施術録、News、報告書ヒント、AI報告書、Drive 保存が強く結び付いている
- この塊を先に触ると影響範囲が広すぎる
- 現段階では「切り出す対象」ではなく「温存して境界を観察する対象」

## 2. 訪問スタッフ勤怠 / 有給

対象外ファイル:

- `src/Code.js` の VisitAttendance 系
- `src/attendance.html`

対象外理由:

- 勤怠、申請、承認、同期、トリガーが一体で動いている
- シート更新と業務ルールが密結合
- 月次表示、申請履歴、管理者承認が同時に存在する

## 3. Albyte 勤怠

対象外ファイル:

- `src/albyte.html`
- `src/albyte_admin.html`
- `src/albyte_report.html`
- `src/Code.js` の Albyte 系

対象外理由:

- スタッフ台帳、勤怠台帳、シフト台帳、給与との関連がある
- テスト用 seed まで同居しており、業務コードと運用コードの分離が未了

## 4. Payroll 全体

対象外ファイル:

- `src/payroll.html`
- `src/payroll_pdf_family.html`
- `src/Code.js` の Payroll 系

対象外理由:

- 従業員、等級、社会保険、源泉、年間集計、PDF 出力まで一体化している
- 影響範囲が大きく、先に請求系やダッシュボード系より触る合理性が薄い

## 5. ルーティング基盤の全面置換

対象外:

- いきなり `doGet` を一本化する作業
- 既存 URL 契約の変更
- 既存テンプレートの統合

対象外理由:

- 入口の正本が複数あり、先に構造分析と運用実態確認が必要
- この段階で手を入れると、対象限定ではなく全面改修になる

## 先に固定すべき前提

再構築に入る前に、最低限次の前提を整理する必要があります。

1. 本番で有効な `doGet` はどれか
2. ダッシュボードと請求が同一 GAS デプロイか、別デプロイか
3. `src/Code.js` のうち、温存対象と将来分離対象をどう線引きするか
4. スプレッドシート ID / Drive ID / Script Properties の正本をどこに置くか

## 推奨する段階的スコープ

### Phase 1

- ダッシュボード系の現状分離境界を固定
- 請求系の境界を固定
- スケジュール候補生成系の境界を固定
- 入口と依存先の対応表を確定

### Phase 2

- ダッシュボード系を単独で再構築
- 請求系を単独で再構築
- スケジュール候補生成系を単独で再構築

### Phase 3

- 残る `src/Code.js` の責務を業務単位で棚卸し
- 施術録本体、勤怠、給与、Albyte を次の候補として再評価

## 明示的にやらないこと

現段階では以下をスコープに含めない。

- リファクタリング
- 命名変更
- 共通化
- 保存方式変更
- 新規機能追加
- 既存ロジック修正
- 既存画面の統合
- シート設計の変更

## 最終判断

最初の再構築対象として最も安全なのは次の順です。

1. ダッシュボード系
2. 請求系
3. スケジュール候補生成系

最初に避けるべきなのは次の順です。

1. `src/Code.js` 全体の置換
2. 施術録本体の再構築
3. 勤怠・給与・Albyte の横断改修

この順序なら、「既存モノリスを壊さず、既に分離され始めている領域から再構築する」という方針を守れます。
